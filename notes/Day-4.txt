Options Pattern:
----------------------
The Options pattern in .NET is a design principle that uses classes to provide strongly-typed access to groups of related settings. 
Instead of reading raw key-value strings from IConfiguration, you bind configuration sections to POCO (Plain Old CLR Object) classes, which are then made available via dependency injection. 

1. Define an Options Class: 
Create a non-abstract class with a public parameterless constructor and public read-write properties that match your configuration keys.

2. Bind in Program.cs: 
Register the class in the DI service container and bind it to a configuration section.

3. Inject and Consume: 
Request an options interface in your service's constructor to access the values via the .Value property

Interface 	Lifetime	Behavior
------------------------------------------------------------------
IOptions<T>		Singleton	Reads configuration once at startup. 
					Does not support runtime changes.

IOptionsSnapshot<T>	Scoped		Recomputed once per request. Ideal for 
	                        	hot-reloading in web apps while 
					maintaining consistency within a single 
					request.

IOptionsMonitor<T>	Singleton	Retrieves the current value at any time. 
					Supports change notifications and is 
					suitable for long-running singleton 
					services.
Advantages:
--------------
1. Type Safety: Avoid "magic strings" and catch errors during compilation instead of at runtime.

2. Validation: Supports Data Annotations (e.g., [Required], [Range]). Using .ValidateOnStart() is a best practice to ensure the app fails immediately if configuration is invalid.

3. Separation of Concerns: Decouples configuration logic from business logic, making code more modular and easier to test with mocks.

4. Named Options: Allows you to bind multiple instances of the same class to different configuration sections (e.g., separate settings for "Email" and "SMS" providers).

Hosting:
-----------------
	1. provide a service provider object (singleton for entire app)
	2. provide a configuration provider object (singleton for entire app)
	3. provide a logger factory object (singleton for entire app) for logger objects
	4. register services with the service provider
	5. configure the configuration provider
	6. configure loggers
	7. manage app start up and shutdown (helps you in graceful shutdown)
	8. manage all of the above objects and their swift disposal
	in short "everything" an app requires

packages:
	Microsoft.Extensions.Hosting
	Microsoft.Extensions.Hosting.Abstractions
